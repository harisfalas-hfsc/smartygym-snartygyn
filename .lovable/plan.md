
# Plan 2 — Two-Cron Split WOD Generation (Final)

Goal: every day at 00:00 Cyprus, both Workouts of the Day (Bodyweight + Equipment) are freshly generated by AI, with full periodization, exercise library matching, image generation, and Stripe products — without ever hitting the 150-second Edge Function timeout.

No library-first fallback at the cron level. Library mode stays as a manual admin toggle only.

---

## What changes

### 1. `generate-workout-of-day` — accept a `slot` parameter

Add an optional input:

```ts
// body: { slot?: 'BODYWEIGHT' | 'EQUIPMENT' | 'VARIOUS' }
```

Behavior:
- If `slot` is provided → generate only that one variant (one AI draft + one normalize/link + one image + one Stripe product). Expected runtime ~60–90s. Safely under 150s.
- If `slot` is omitted → keep current behavior (both variants, used only by manual admin runs / "Generate Now" button).
- Periodization (date in Cyprus tz, category, difficulty, format) is computed identically in both calls so the two slots stay coherent for the same day.
- All existing code paths stay: library-first selection, exercise density validation, normalization, Stripe product/price creation, idempotency, image generation, atomic publish per variant.

No removal of any feature. Just a scope filter.

### 2. Cron schedule — two jobs instead of one

Replace the single midnight job with two staggered jobs (Cyprus time → UTC):

| Job name | Time (Cyprus) | Time (UTC, summer/winter handled by fixed UTC) | Slot |
|---|---|---|---|
| `generate-wod-bodyweight-daily` | ~00:05 | `5 21 * * *` (winter) / configurable | BODYWEIGHT |
| `generate-wod-equipment-daily` | ~00:25 | `25 21 * * *` | EQUIPMENT |

20-minute gap = even if Bodyweight runs the full 150s, Equipment still starts cleanly on a fresh invocation. Each cron POSTs `{ "slot": "BODYWEIGHT" }` or `{ "slot": "EQUIPMENT" }` to `generate-workout-of-day`.

Old single job (`generate-wod-daily` / `generate-workout-of-day`) is unscheduled.

### 3. Restore fresh-generation as the default

- Set `wod_auto_generation_config.wod_mode = 'generate'`.
- The library-first path (`select-wod-from-library`) is **only** invoked when an admin explicitly switches mode in the admin panel. Never by cron.

### 4. Watchdog & backup — slot-aware retries

- `watchdog-wod-check`: at a safe time after both crons (e.g. 01:30 Cyprus), check which of today's slots is missing in `admin_workouts` for `generated_for_date = today (Cyprus)`. If Bodyweight missing → call generator with `{slot:'BODYWEIGHT'}`. Same for Equipment. Never trigger both at once. Never silently switch to library mode.
- `backup-wod-generation`: same logic — targeted per slot, never both, never library fallback.

### 5. Admin "Generate Now" button

Stays as-is (no slot param) → produces both variants in one call. Used for manual recovery only. Acceptable because admin can wait/retry; cron path is what must be reliable.

### 6. Admin panel toggle (already exists)

The `wod_mode` toggle in admin remains the only place library mode can be turned on. Cron never overrides it back to library. Default after this change: `generate`.

---

## Why this is safe

- Each cron call processes exactly ONE workout end-to-end → well under 150s.
- Zero new tables, zero new orchestrator code, zero state machine. Only:
  - one new optional parameter,
  - two cron entries instead of one,
  - small filter in watchdog/backup.
- All existing safeguards (library-first exercise selection, density validation, Stripe idempotency, atomic publish, image trigger) keep working unchanged.
- Failure of one slot does not affect the other. Watchdog repairs only the missing one.
- If a generation legitimately fails (e.g. AI gateway 500), it fails loudly — no silent switch to library mode.

3. **Migration: restore fresh mode**

   - Detect missing slot(s) for today (Cyprus date).
   - Invoke generator per missing slot with `{slot:'BODYWEIGHT'|'EQUIPMENT'}`.
